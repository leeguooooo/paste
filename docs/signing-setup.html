Pastyx macOS 签名与公证配置手册

从零开始：申请 Apple Developer ID 证书 → 配置 GitHub Secrets → 发布签名 + 公证（Notarized）版本 → 启用自动更新（auto-update）。

适用范围：.github/workflows/release.yml · apps/macos/electron-builder.json · electron-builder 26 · electron-updater 6

现状与目标 Current behavior vs. after setup

现在（未配置 Secrets）配置 Secrets 之后
代码签名 electron-builder 跳过签名（CSC_IDENTITY_AUTO_DISCOVERY=false），afterPack.cjs 对 .app 做 ad-hoc 签名兜底 使用 Developer ID Application 证书 + Hardened Runtime + entitlements 正式签名
公证 Notarization 跳过 electron-builder 自动提交 Apple 公证并 staple 票据
Gatekeeper 用户首次打开需手动放行（右键打开 / 系统设置） 双击直接打开，无警告
自动更新 托盘菜单显示 "Auto-update unavailable"，引导用户去 GitHub Releases 手动更新 托盘 "Check for Updates..." 可用，electron-updater 全自动更新
CI 验证 签名 / 公证校验步骤自动跳过，流水线照常出包 CI 强制校验 Developer ID Application 签名链 + stapler validate，不达标即失败
设计原则：所有签名相关步骤都以「Secret 是否非空」为条件。不设置任何 Secret 时，发布流程与今天完全一致，零破坏。空字符串 Secret 永远不会泄漏给 electron-builder（参见提交 7ca2f5d）。

操作步骤 Step-by-step

1加入 Apple Developer Program（$99 / 年）

  1. 访问 https://developer.apple.com/programs/enroll/，用你的 Apple ID 登录并以个人（Individual）身份注册，支付 $99/年。
  2. 审核通过后，在 https://developer.apple.com/account 的 Membership details 页面记下 Team ID（10 位字母数字，如 A1B2C3D4E5）——之后要填进 APPLE_TEAM_ID。

2创建 "Developer ID Application" 证书

  1. 在 Mac 上打开 钥匙串访问（Keychain Access） → 菜单「证书助理」→「从证书颁发机构请求证书…」，填邮箱、选「存储到磁盘」，生成 CertificateSigningRequest.certSigningRequest（CSR）。私钥会留在你这台 Mac 的钥匙串里。
  2. 打开 https://developer.apple.com/account/resources/certificates/add，证书类型选 Developer ID Application（注意：不是 "Apple Development"，也不是 "Developer ID Installer"），上传 CSR，下载 developerID_application.cer。
  3. 双击 .cer 导入钥匙串。在「我的证书（My Certificates）」分类下应能看到 Developer ID Application: 你的名字 (TEAMID)，且左侧有展开箭头（说明私钥已配对）。
关键：证书必须出现在「我的证书」分类且能展开看到私钥。如果只在「证书」分类里、不能展开，说明这台 Mac 没有对应私钥（CSR 不是在这台机器生成的），导出的 .p12 将无法用于签名。

3导出 .p12

  1. 钥匙串访问 →「我的证书」→ 右键 Developer ID Application: … →「导出…」，格式选 个人信息交换 (.p12)。
  2. 设置一个导出密码（这就是 MACOS_CERTIFICATE_PASSWORD / CSC_KEY_PASSWORD），保存为 developer-id.p12。
  3. 导出时选中证书那一行（会连同私钥一起导出），不要只展开后选私钥。

4base64 编码

base64 -i developer-id.p12 | pbcopy   # 结果已复制到剪贴板

这串 base64 文本就是 MACOS_CERTIFICATE（即 electron-builder 的 CSC_LINK）的值。

5生成 App 专用密码（App-Specific Password）

  1. 访问 https://account.apple.com →「登录与安全」→「App 专用密码」→ 生成，命名如 pastyx-notarize。
  2. 得到形如 abcd-efgh-ijkl-mnop 的密码，即 APPLE_APP_SPECIFIC_PASSWORD。注意：这不是你的 Apple ID 登录密码。

6设置 GitHub Repo Secrets

仓库 → Settings → Secrets and variables → Actions → New repository secret，逐个添加：

Secret 名称值必需
MACOS_CERTIFICATE步骤 4 的 base64 字符串（workflow 也接受别名 CSC_LINK，二者设其一即可）是
MACOS_CERTIFICATE_PASSWORD步骤 3 的 .p12 导出密码（别名 CSC_KEY_PASSWORD）是
APPLE_ID你的 Apple ID 邮箱公证必需
APPLE_APP_SPECIFIC_PASSWORD步骤 5 的 App 专用密码公证必需
APPLE_TEAM_ID步骤 1 的 Team ID公证必需

三个 APPLE_* 公证 Secret 必须同时设置；只设一部分时 CI 会在打包前直接报错提示（fail fast），不会产出半成品。

备选方案：workflow 同时支持 App Store Connect API Key 方式公证：APPLE_API_KEY（.p8 文件内容）+ APPLE_API_KEY_ID + APPLE_API_ISSUER。两种方式选一种即可，Apple ID 方式更简单。

7触发发布

git tag v0.3.5
git push origin v0.3.5

release.yml 由 v* tag 触发。配好 Secrets 后无需改任何代码 / 配置——签名与公证自动生效。

8验证

CI 端（workflow 自动执行，失败即整个 release 失败）：

  • "Verify Developer ID signature…" 步骤：codesign -dv --verbose=4 必须输出 Authority=Developer ID Application: …
  • "Verify notarization…" 步骤：xcrun stapler validate Pastyx.app 通过

本地端（下载 Release 的 dmg 安装后）：

codesign -dv --verbose=2 /Applications/Pastyx.app
# 期望: Authority=Developer ID Application: 你的名字 (TEAMID)
#       flags=0x10000(runtime)   ← Hardened Runtime 已启用

spctl -a -vv /Applications/Pastyx.app
# 期望: accepted  source=Notarized Developer ID

App 内：启动 Pastyx，点菜单栏托盘图标——更新菜单项应显示可点击的 "Check for Updates..."，而不是灰色的 "Auto-update unavailable"。从下一个签名版本开始，应用即可全自动升级。

故障排查 Troubleshooting

症状原因处理
electron-updater 报 code signature ... did not pass validation，托盘弹窗提示去 GitHub Releases 手动更新 当前已安装的版本是未签名 / ad-hoc 签名的旧包；Squirrel.Mac 要求新旧版本签名链一致 预期行为。让用户从 GitHub Releases 手动安装第一个签名版本，之后的更新即可全自动
CI "Verify Developer ID signature" 步骤 grep 不到 Authority=Developer ID Application: .p12 里装的是 "Apple Development" 等错误类型证书，或 p12 缺私钥导致 electron-builder 静默回退 回到步骤 2/3 重新签发并导出，确认证书类型为 Developer ID Application 且从「我的证书」导出
构建日志出现 Env WARNING: CSC_LINK ... cannot be decoded 或把仓库目录当证书路径 base64 串包含换行 / 被截断，或 Secret 值为空字符串 用 base64 -i file.p12（macOS 默认不换行）重新生成；确认 Secret 粘贴完整无空白
公证失败：HTTP status code 401 / Invalid credentials APPLE_APP_SPECIFIC_PASSWORD 填了 Apple ID 登录密码，或 APPLE_TEAM_ID 与证书 Team 不一致 重新生成 App 专用密码；核对 Membership 页面的 Team ID 与证书括号里的 TEAMID 一致
公证返回 Invalid，日志含 The executable does not have the hardened runtime enabled hardenedRuntime / entitlements 配置被改掉 确认 electron-builder.json 中 mac.hardenedRuntime: true 且 build/entitlements.mac*.plist 两个文件存在
xcrun stapler validate 失败但公证显示成功 staple 需要联网回查 Apple CDN，偶发传播延迟 等几分钟重跑该 job；若持续失败，本地 xcrun stapler staple Pastyx.app 看详细错误
导出 .p12 时「个人信息交换 (.p12)」选项是灰色的 选中的是「证书」分类下没有私钥的副本 切换到「我的证书（My Certificates）」分类再导出；若那里没有，说明私钥在别的机器上，需在生成 CSR 的那台 Mac 上操作
只设置了部分 APPLE_* Secret，CI 在 "Validate Apple ID notarization secrets" 步骤失败 三个公证 Secret 必须成组出现，且必须同时配置签名证书 预期的 fail-fast。补齐 APPLE_ID + APPLE_APP_SPECIFIC_PASSWORD + APPLE_TEAM_ID + MACOS_CERTIFICATE，或全部删掉回到无签名模式

背景：为什么 macOS 自动更新依赖签名 Why signing gates auto-update

electron-updater 在 macOS 上基于 Squirrel.Mac，它强制校验更新包与已安装应用的代码签名链。未签名 / ad-hoc 签名的包无法通过校验，所以 apps/macos/electron/auto-update.cjs 主动检测：只有当前二进制带有受信任的 Developer ID Application: 签名时才启用自动更新，否则提示走 GitHub Releases 手动更新。这段逻辑是正确的保护，配好证书后它会自动放行，无需改代码。
